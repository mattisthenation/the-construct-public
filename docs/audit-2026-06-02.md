# The Construct — Production Audit (2026-06-02)

Exhaustive review across four dimensions (correctness/robustness, production/ops,
DX/config/docs, security) of `main` after Slice 3 merged. Findings are evidence-based
(file:line) and ranked by severity with a confidence score. The final section splits them
into **Fixed in this pass** vs **Recommended follow-ups**.

The design is sound — deterministic-first gates contain the LLM well, crash recovery
(claim + reconcile) works for the common case, and per-note serialization is correct.
The issues below are about making it robust and operable for always-on / fresh-machine use.

---

## Critical

### A. No `tracing` subscriber is ever installed — all logs are silently discarded
`crates/construct-cli/src/main.rs` / `commands.rs` never call `tracing_subscriber`. Every
`tracing::error!/warn!/debug!` in the codebase (handler errors, "inbox skipped — no agent",
"invalid daily_time", "daily summary failed", reconcile warnings) goes into the void.
**Scenario:** Ollama is down for an hour, dozens of runs fail and stamp `construct_status:
error` on notes — and nothing is printed anywhere. You're flying blind. (confidence 98)
**Fix:** install `tracing_subscriber::fmt().with_env_filter(...)` in `main`. — *fixed this pass.*

### B. No HTTP timeout on any client → a hung Ollama/URL stalls a run forever and holds the per-note lock
`reqwest::Client::new()` with no timeout in `construct-model-ollama/src/lib.rs:16`,
`web_fetch.rs:14`, `web_search.rs:16`; `run_loop` awaits with no deadline.
**Scenario:** Ollama accepts the connection but never responds (model loading, swap thrash,
network black-hole after laptop wake) → the handler hangs forever holding that note's
`AsyncMutex`; the note is stuck `queued/running` and later events for it block silently.
(confidence 88) **Fix:** `Client::builder().timeout(...).connect_timeout(...)`. — *fixed this pass.*

### C. Spawned long-lived tokio tasks can panic and die silently, permanently disabling a subsystem
The forwarder, idle poller, scheduler, and per-event handlers are bare `tokio::spawn`s whose
`JoinHandle`s are dropped (`watch_loop.rs`). A panic (e.g. an `unwrap()` on an odd path)
kills that task for the life of the process with no log and no restart. If the **forwarder**
dies, the file watcher stops feeding events while the process still looks healthy.
(confidence 90) **Fix:** supervise tasks (respawn or abort-the-process so a supervisor
restarts) and log on exit. — *recommended follow-up.*

### D. `construct.db` is opened at a CWD-relative path → split-brain
`watch_loop.rs:62` and `commands.rs:115` hardcode `"sqlite://construct.db"` with
`create_if_missing`. Launching from a different directory silently creates a *new empty*
DB: reconcile recovers nothing, `runs`/`status` show nothing while `watch` writes elsewhere.
(confidence 95) **Fix:** resolve an absolute path (relative to the config file or an
XDG/`~/.local/share` dir); print it on startup. — *banner + "run from install dir" doc this
pass; full path-resolution = recommended follow-up.*

### E. `api_key_env` foot-gun — the live config pasted the Tavily **key** where the env-var **name** goes (and the key is now exposed)
Code does `std::env::var(&ws.api_key_env)` — `api_key_env` must be the *name* of an env var.
The real `construct.toml` set it to the literal `tvly-…` key, so `std::env::var("tvly-…")`
fails, `.unwrap_or_default()` swallows it, and web search silently runs with an empty key.
(confidence 99) **Action for you:** **rotate that Tavily key** (it was in a local file read
during the audit), set the field to a name like `api_key_env = "TAVILY_API_KEY"`, and
`export TAVILY_API_KEY=…` (or a shell profile). **Fix:** add config validation that rejects
a key-shaped value + warn at startup when the env var is empty. — *sample config clarified +
validation = recommended follow-up.*

## Important

### F. No retry/backoff — a transient Ollama blip becomes a permanent note error
On any model error the pipeline calls `fail()` → stamps terminal `construct_status: error`.
Because `classify()` only triggers on notes with *no* status, an error from a 5-second blip
is **sticky**: the note is never retried automatically; you must clear the status by hand.
(confidence 90) **Fix:** distinguish transient transport errors from gate/validation errors;
retry transient with backoff and/or don't stamp a terminal status for them. — *recommended.*

### G. `system_prompt_file` is CWD-relative and fails silently
`prompts/scout.md` etc. are read relative to the process CWD; run `watch` from elsewhere and
the read returns `Err`, `.ok()` swallows it, and the agent silently uses a generic built-in
prompt. (confidence 95) **Fix:** resolve relative to the config file's directory; warn on a
missing configured prompt instead of silent fallback. — *tarball ships `prompts/` + "run from
install dir" doc this pass; config-relative resolution = recommended follow-up.*

### H. `init` / `SAMPLE_CONFIG` is stale — no `[inbox]`/`[journal]`/`[schedule]`, LAN `base_url`, likely-invalid model tag
`SAMPLE_CONFIG` omits all Slice 3 tables, uses `base_url = http://192.168.1.33:11434`
(a LAN IP) and `qwen3.6:27b` (no such Qwen tag). A fresh `init` → `watch` hangs/fails with
no hint. (confidence 96/70) **Fix:** add commented Slice 3 blocks, `localhost` base_url,
a real model tag. — *fixed this pass.*

### I. Web fetch SSRF — no scheme/host restriction
`web_fetch.rs` GETs any URL from note content, following redirects, with no host filtering:
`http://127.0.0.1:11434` (Ollama), `http://192.168.x` LAN admin panels, `169.254.169.254`
metadata. Blast radius on a personal Mac is modest (responses only flow into the local model
+ a note), but a malicious note/page could probe internal services. (confidence 90)
**Fix:** reject loopback/private/link-local after DNS resolution; `redirect::Policy::limited`;
re-check each hop. — *recommended follow-up.*

### J. Web fetch buffers the whole response body before truncating → OOM risk
`resp.text().await` reads the entire body into memory, then trims to 8000 chars. A multi-GB
or infinite-stream response can OOM the daemon. (confidence 85) **Fix:** cap bytes via
`bytes_stream()` or `Content-Length`. — *recommended (timeout added this pass helps).*

### K. Inbox idle poller re-emits a note every 60s during a long run (duplicate queued runs)
`run_inbox` writes the note once at the end, so for the whole multi-minute run the on-disk
file has no `construct_status`; every 60s tick re-emits `IdleNote`. The per-note lock
serializes them (no concurrency/corruption) but duplicate claims/runs stack up behind the
lock, each doing full LLM work. (confidence 68) **Fix:** stamp `construct_status: queued` to
disk at the *start* of `run_inbox` (like the tag pipeline), or track in-flight paths. —
*recommended follow-up.*

### L. Inbox auto-move can lose enrichment on a crash between `rename` and `write`
`run_inbox` renames the note to the target, then writes the finalized (enriched) content. The
enrichment lives only in memory until that write. A crash in between leaves the moved file
with pre-enrichment content and the run stuck (and reconcile points at the now-missing
original path). (confidence 72) **Fix:** write enriched content to the original path *before*
renaming, or write-temp-then-rename atomically. — *recommended follow-up.*

### M. `MissedTickBehavior::Burst` after sleep → backlogged ticks fire back-to-back on wake
Both pollers use default `tokio::time::interval` (Burst). After an 8-hour sleep the idle
poller and scheduler fire many ticks at once. `due()` gates the daily run correctly, but if
`set_last_run` ever fails (DB busy during the burst) the summary re-fires every 60s.
(confidence 80) **Fix:** `ticker.set_missed_tick_behavior(Skip)` on both; handle the
`set_last_run` error. — *fixed this pass (Skip); error-handling = follow-up.*

### N. README has no Ollama prerequisites and no Slice 3 user docs
README covers Slices 1–2 only; nothing about installing/running Ollama, pulling models, or
the inbox/daily/schedule features. A fresh-machine setup following the README gets silent
failures. (confidence 95) **Fix:** add Ollama prereqs + a Slice 3 section. — *fixed this pass.*

## Minor (recommended)

- **`run_for_note` ordering uses 1-second `created_at` precision** (`store/lib.rs:98`,
  `migrations/0001_init.sql`) — two runs in the same second can be mis-ordered, so
  idempotency/decision can act on a stale run. Fix: `ORDER BY created_at DESC, rowid DESC` or
  sub-second timestamps. (confidence 75)
- **Per-note lock `HashMap` grows unbounded** for the process lifetime (`watch_loop.rs`
  `lock_for`). Slow leak over weeks. Fix: evict when `Arc::strong_count == 1`. (confidence 85)
- **Repeated malformed JSON has no repair/retry** — one bad local-model completion fails the
  whole run; the 8-iteration budget is only spent on tool calls. Fix: on gate `NotJson`,
  re-prompt "your output wasn't valid JSON" within budget. (confidence 65) *(This was also on
  the carried-over backlog as `extract_json` repair/retry.)*
- **Non-atomic note writes everywhere** (`std::fs::write`) — a crash mid-write truncates a
  note. Fix: write-temp-then-rename for all note writes. (confidence 45)
- **`StatusChanged` broadcast handler swallows errors** (`let _ = o.handle(...)`) with no log,
  unlike the tag/inbox branches. (confidence 70)
- **`status` and `runs` are identical** and print nothing on an empty result. Differentiate;
  add a `No runs yet.` message. (confidence 90)
- **Whole-vault rescan per run** (`existing_tags`, `list_folders`, `walk_notes`) — O(vault)
  cost per note; inbox does both per note + the poller reads every top-level inbox file every
  60s. Fine for small vaults; add an mtime/watcher-invalidated cache for large ones.
  (confidence 80)
- **No graceful shutdown** (SIGTERM/SIGINT) — an in-flight handler is killed mid-pipeline.
  Reconcile recovers most cases. Fix: drain in-flight tasks on signal. (confidence 70)
- **`changed_notes_on` uses last-mtime** — a note edited yesterday *and* today is attributed
  to today and dropped from yesterday's summary; near-midnight/DST skew can shift a note's
  day. Acceptable; document, or use run history instead of mtime. (confidence 62)
- **File perms 0644** on `construct.db`/`construct.toml` — neither leaks the key today, but
  `0600` is cheap defense-in-depth on a shared machine. (confidence 80)

## What the security review confirmed is *already safe*
- The LLM **cannot move files outside the vault**: organize `validate_organize` and inbox's
  live `folders.contains(dest)` check both constrain destinations to real scanned folders
  (folder names can't contain `..`). Hand-editing `construct_proposed_move` to a traversal
  path is the only escape — self-inflicted, local, low severity (defense-in-depth: re-check
  on accept).
- The Tavily key is **never written to `construct.db` or notes** — it only goes in the Tavily
  HTTPS body; `construct.toml` holds the env-var *name*, not the value.
- Research **fabricated-source injection is blocked** — `validate` rejects any cited URL not
  present in gathered evidence.
- Prompt-injection residual risk is **data-quality** (mis-tag/mis-file/junk summary behind
  human review), not vault escape or code execution. The deterministic-first design works.

## Summary: Fixed in this pass vs. Recommended follow-ups

**Fixed in this pass (this branch):**
- A — initialize `tracing_subscriber` (set `RUST_LOG`, default `info`).
- B — HTTP timeouts on Ollama / web_fetch / web_search clients.
- M — `MissedTickBehavior::Skip` on the idle + schedule pollers.
- H — refresh `SAMPLE_CONFIG` (Slice 3 tables, `localhost` base_url, valid model tag).
- D/G (partial) — startup banner prints vault path, DB path, and active automations; the
  release tarball ships `prompts/` + config and the setup doc says "run from the install dir."
- N — README: Ollama prerequisites + a Slice 3 / automations section.

**Recommended follow-ups — NOW IMPLEMENTED (second hardening pass on this branch):**
1. ✅ C — forwarder/idle/scheduler loop bodies wrapped in `catch_unwind` (panic → log +
   continue); panic-prone path `unwrap()`s in the move paths guarded.
2. ✅ F — `chat_with_retry` retries `ModelError::Transport` (Ollama blips) up to 3× with backoff.
3. ✅ D/G — `construct.db` + `system_prompt_file` resolved relative to the config dir
   (`resolve_db_url`/`load_system_prompt`); warn on a missing prompt file.
4. ✅ I/J — web-fetch SSRF guard (reject loopback/private/link-local, screen redirect hops)
   + 2 MB streamed response cap.
5. ✅ E — `validate()` rejects a key-shaped `api_key_env`; `config-check` reports whether the
   env var resolves.
6. ✅ L — organize-accept + inbox auto-move now write-then-rename (crash leaves a recoverable
   note). K was found to be already safe (the claim stamps `queued` to disk before the long run).
7. ✅ Minors — `created_at, rowid` tiebreaker; lock-map pruning; broadcast error logging;
   `status`/`runs` split; atomic note writes everywhere; graceful Ctrl-C shutdown; 0600 DB
   perms; `MissedTickBehavior::Skip`; `changed_notes_on` mtime limitation documented.

**Still deferred (one item):** the whole-vault rescan cache (perf optimization for large
vaults) — needs its own design (incremental, watcher-invalidated index) and carries
correctness risk, so it's intentionally left for a dedicated change. Also still open from the
original backlog: chat-TUI async/streaming send, and a full LLM JSON-repair re-prompt (the
gate already tolerates prose-wrapped JSON; transient retry covers transport blips).

**Immediate action for you:** rotate the Tavily API key and set `api_key_env` to an env-var
*name* (e.g. `TAVILY_API_KEY`), then export that variable. (Finding E.) Note: with the new
validation, your current `construct.toml` (key pasted into `api_key_env`) will now fail
`config-check`/`watch` with a clear message until you fix it this way.
